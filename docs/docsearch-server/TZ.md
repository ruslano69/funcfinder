# ТЗ: docsearch-server — сервер документации для команд с распределённой разработкой

Статус: черновик v0.1 · 2026-07-01
Источник: сессия проектирования (funcfinder / docsearch)

---

## 1. Назначение

**docsearch-server** — это единый *источник правды* для команды: сервер, в
котором живут спецификации, ТЗ, документация библиотек, планы спринтов, решения
и changelog, а результаты выполненной работы возвращаются обратно. Он служит
**слоем грундинга** для LLM-агентов и людей: и те и другие принимают решения,
опираясь на одну актуальную, версионированную и воспроизводимую правду.

Ключевая ценность — сделать модель (и человека) компетентной в *текущем*
состоянии проекта и его библиотек, включая нишевые и быстро меняющиеся API,
**без дообучения** — за счёт извлечения из общей базы, а не из претрейна.

## 2. Проблема

- LLM-агенты грундятся на устаревших претрейн-знаниях; на нишевых/свежих API
  галлюцинируют.
- Знания команды размазаны по вики, чатам, тикетам, головам — нет единой точки
  правды, на которую можно сослаться воспроизводимо.
- При распределённой разработке (люди + рой агентов) правда «едет под ногами»:
  спека поменялась посреди спринта, и разные участники грундятся на разном.
- Векторные БД (Pinecone/Qdrant/Chroma) — это инфраструктура, которую надо
  провиженить и обслуживать; они хранят векторы, но не решают задачу *командной
  правды с обратной связью*.

## 3. Целевая аудитория

- Команды, работающие с LLM-агентами в разработке (single-agent и мульти-агент).
- Проекты на нишевых/быстро меняющихся стеках, где грундинг на свежих доках
  критичен.
- Команды с распределённой разработкой, которым нужна воспроизводимая база правды.

## 4. Цели и не-цели

### Цели
1. Единая версионированная база правды на команду/проект.
2. Гибридное извлечение (полнотекст + вектор + regex) для грундинга агентов и людей.
3. Петля обратной связи: результаты работы (changelog, готовые задачи, решения)
   возвращаются в базу и повышают её ценность со временем.
4. **Воспроизводимый грундинг**: обращение к именованной версии правды всегда
   даёт идентичный результат.
5. Zero-infra: разворачивается без внешних сервисов, единый бинарник, единый файл
   на релиз.
6. Первоклассный интерфейс для LLM — **MCP-сервер**.

### Не-цели (сознательные ограничения)
- **НЕ система управления задачами.** Не заменяет Jira/Linear/GitHub Issues, не
  ведёт канбан, не назначает исполнителей. docsearch-server — **шина правды, а не
  начальник**. Распределение задач: участники *читают* задачи своей роли из слоя
  правды и *пишут* результат обратно; маршрутизацию/трекинг делает внешний
  оркестратор (человек или агент) или интеграция с существующим трекером.
- НЕ конкурент векторным БД по масштабу (миллиарды векторов, шардинг по кластеру).
  Ниша — командная правда рядом с кодом, а не хранилище планетарного масштаба.
- НЕ считает эмбеддинги сам (BYO-эмбеддинги, см. §7.2).

## 5. Глоссарий

| Термин | Значение |
|---|---|
| **Источник правды** | Версионированный набор документов, отражающий актуальное состояние проекта. |
| **rewrite-сторона** | Путь записи: ингест и петля обратной связи. Один сериализованный писатель. |
| **readonly-сторона** | Путь чтения: грундинг. Пул читателей на неизменяемом снапшоте. |
| **release (релиз)** | Неизменяемая именованная версия правды (напр. `truth-2026.07`). Не меняется после публикации. |
| **freeze (заморозка)** | Окно стабилизации перед релизом; аналог freeze-периода дистрибутива. |
| **channel (канал)** | Указатель на релиз: `stable` / `testing` / `unstable`. |
| **pin (пиновка)** | Привязка потребителя к конкретному релизу/каналу. |
| **провенанс** | Метаданные записи: автор, время, против какой версии правды сделано. |
| **грундинг** | Извлечение релевантной правды в контекст агента/человека. |

## 6. Функциональные требования

### 6.1 Ингест (rewrite)
- FR-1. Приём документов типов: `spec`, `ТЗ`, `lib_doc`, `sprint`, `changelog`,
  `task`, `decision`, `general`.
- FR-2. Форматы входа: `.md`, `.txt`, `.pdf` (с чанкингом по секциям + overlap),
  а также структурированные записи (JSON) для changelog/задач.
- FR-3. Каждый документ несёт метаданные: автор, время, роль(и), путь/источник,
  версия-источника.
- FR-4. PDF проходит quality-gate (плохие сканы отклоняются).

### 6.2 Петля обратной связи (rewrite)
- FR-5. Действие `record`: возврат результата работы в базу — changelog-запись,
  закрытая задача, принятое решение — с провенансом и ссылкой на исходную задачу/спеку.
- FR-6. Накопленные записи попадают в **следующий** релиз, не мутируя опубликованный.

### 6.3 Извлечение (readonly)
- FR-7. Гибридный поиск: FTS5 (полнотекст) + векторный (cosine/L2) + regex, режим
  `hybrid` с корректной деградацией до FTS при отсутствии эмбеддингов.
- FR-8. `recall` — извлечение по теме с ранжированием.
- FR-9. `context(role)` — представление правды под конкретную роль (view-фильтр по
  тегам ролей), одна база, разные линзы.
- FR-10. Любой запрос выполняется **против конкретного релиза** (по умолчанию —
  релиз, на который указывает канал потребителя).

### 6.4 Релизы и каналы (publish)
- FR-11. `freeze` — открыть окно стабилизации (в testing перестают литься новые
  изменения; правки идут в unstable).
- FR-12. `publish` — атомарно материализовать текущее состояние в неизменяемый
  релиз `truth-YYYY.MM[.patch]`.
- FR-13. Каналы `stable` / `testing` / `unstable` — указатели на релизы; смена
  указателя атомарна («день релиза»).
- FR-14. Point-release: срочная правка публикуется как новый релиз
  (`truth-2026.07.1`), не мутируя предыдущий; потребители опт-инятся.
- FR-15. Хранить последние K релизов для time-travel/аудита; старьё
  архивируется/чистится по политике.

### 6.5 Провенанс и воспроизводимость
- FR-16. Обращение к релизу `R` всегда возвращает идентичный результат (пока `R`
  хранится).
- FR-17. По любой записи можно узнать: автор, время, против какого релиза сделано.
- FR-18. Возможность запросить «дифф правды» между двумя релизами.

### 6.6 Интерфейсы
- FR-19. **MCP-сервер** — основной интерфейс для агентов (см. §10.1).
- FR-20. Сетевой API (TCP/HTTP) для read-серверов.
- FR-21. CLI для эпизодических операций и администрирования.

## 7. Архитектура

### 7.1 Разделение readonly / rewrite (CQRS)

```
     REWRITE side (истина втекает)            READONLY side (грундинг)
  ┌────────────────────────────────┐     ┌──────────────────────────────┐
  specs / ТЗ / доки / решения             рой агентов + люди
  changelog / выполненные задачи ─►         ▲ ▲ ▲ ... (реплики, N узлов)
        │                                    │
  writer-демон:                         read-серверы (stateless):
   • один сериализованный писатель       • пул читателей на снапшоте
   • транзакции, батч-запись             • SQLITE_OPEN_READONLY + query_only
   • провенанс + версии                  • канал → релиз, hot-swap
        │                                    ▲
        ▼          ── publish ──►            │
  live write-log DB ───────────►  truth-2026.07.sqlite (immutable)
                                  truth-2026.06.sqlite … (история)
```

- **Контеншн исчезает по конструкции**: читатели не касаются пишущей БД.
- **Read-сторона масштабируется репликами**: релиз — неизменяемый файл, копируется
  на N stateless-узлов; балансировщик распределяет запросы.
- **publish** = `VACUUM INTO 'truth-YYYY.MM.sqlite'` (или backup-API) — атомарный
  слепок с уже запечёнными эмбеддингами. Read-серверы горячо перецепляются на новый
  указатель канала.
- **Write-путь** = батч-запись одним писателем (WAL, `synchronous=NORMAL`,
  `busy_timeout`), без гонки с чтением.

### 7.2 BYO-эмбеддинги
- Векторы вычисляются снаружи (локальный Ollama, напр. `qwen-embedding:0.6`, или
  любой провайдер) и передаются в ингест. Сервер их **хранит и сравнивает**, но не
  считает. Провайдер-агностик; размерность конфигурируется.

### 7.3 Каналы как указатели
```
stable   → truth-2026.07
testing  → truth-2026.08-rc1
unstable → (live write-log)
```
Потребитель пинится к каналу или к конкретному релизу. Продакшн-агенты → `stable`,
агенты на передовой → `testing`.

## 8. Модель данных (эскиз)

Внутри одного релиза (`truth-*.sqlite`):

```sql
-- документы правды
documents(
  id INTEGER PRIMARY KEY,
  source_type TEXT,        -- spec|ТЗ|lib_doc|sprint|changelog|task|decision|general
  title TEXT,
  path TEXT,
  role_tags TEXT,          -- CSV/JSON ролей для view-фильтра
  author TEXT,
  created_at INTEGER,
  source_version TEXT      -- провенанс: версия исходника
);

-- чанки + эмбеддинги
chunks(
  id INTEGER PRIMARY KEY,
  document_id INTEGER REFERENCES documents(id),
  ordinal INTEGER,
  body TEXT,
  embedding BLOB           -- вектор (BYO), NULL если без эмбеддинга
);

CREATE VIRTUAL TABLE chunks_fts USING fts5(body, content='chunks', content_rowid='id');
```

Контрольная БД (у writer-демона, вне релизов):

```sql
releases(version TEXT PRIMARY KEY, created_at INTEGER, frozen_at INTEGER,
         parent TEXT, notes TEXT);
channels(name TEXT PRIMARY KEY, release_version TEXT REFERENCES releases(version));
provenance(record_id INTEGER, author TEXT, ts INTEGER, target_release TEXT,
           source_ref TEXT);   -- связь результата с исходной задачей/спекой
```

## 9. Нефункциональные требования

- NFR-1. **Производительность чтения.** Ориентир (измерено на прототипе, Go +
  modernc, 8 ядер, localhost): **~23k req/s на узел** при sub-мс латентности до
  насыщения. Масштабирование — линейно репликами read-узлов.
- NFR-2. Чистый запрос (FTS5 MATCH + вычитка результата) — порядка ~100 мкс;
  основной оверхед — слой `database/sql` (Scan-рефлексия), рычаг оптимизации —
  прямой драйвер при необходимости.
- NFR-3. **Zero-infra**: единый cgo-free бинарник, кросс-компиляция под
  linux/darwin/windows (amd64/arm64) без тулчейнов; данные — файлы SQLite.
- NFR-4. **Консистентность**: read-сторона eventually consistent с rewrite (лаг
  публикации) — это *свойство*, а не дефект: публикуются когерентные версии, не
  полу-записанное состояние.
- NFR-5. **Immutability**: опубликованный релиз неизменяем; коррекции — только новые
  релизы.
- NFR-6. **Безопасность/доступ**: роли как view-фильтр; (v2) аутентификация
  read/write путей, разграничение кто может `publish`.
- NFR-7. **Наблюдаемость**: метрики rps/латентности/глубины очереди записи, лог
  публикаций.

## 10. Интерфейсы

### 10.1 MCP-инструменты
Разложены строго по сторонам CQRS:

**Readonly (грундинг):**
- `search(query, mode=hybrid, limit, release?)` — гибридный поиск.
- `recall(topic, limit, release?)` — извлечение по теме.
- `context(role, release?)` — правда под роль.
- `get_release(name)` / `list_releases()` / `diff_releases(a, b)`.

**Rewrite (истина внутрь):**
- `ingest(doc, type, role_tags, source_version)` — приём документа.
- `record(result, source_ref, author)` — возврат результата (changelog/задача/решение).
- `freeze(channel=testing)` / `publish(name, notes)` — релизный цикл.
- `set_channel(name, release)` — перецепка канала (админ).

Каждый агент в конфиге указывает канал/релиз, к которому пинится.

### 10.2 CLI
`docsearch-server serve|ingest|record|freeze|publish|channels|search` — тонкая
обёртка над теми же операциями для людей и CI.

### 10.3 Сетевой API
Read-серверы: TCP/HTTP, `TCP_NODELAY`, keep-alive, персистентный пул соединений.

## 11. Технологический стек

- **Go** (единый бинарник, native-async: горутины + каналы = очередь+семафор без
  poll-loop).
- **modernc.org/sqlite** — cgo-free, тот же движок SQLite (транслированный C),
  FTS5 включён, кросс-компиляция без боли.
- **SQLite WAL** на write-стороне; неизменяемые файлы-релизы на read-стороне.
- **MCP** — интерфейс для LLM-агентов.

## 12. Этапы

**MVP (v0):**
- writer-демон с ингестом + `record`, WAL, батч-запись, провенанс.
- `publish` = `VACUUM INTO` в именованный релиз; каналы-указатели.
- один read-сервер (TCP), гибридный поиск на снапшоте, hot-swap канала.
- MCP-обёртка: `ingest`, `record`, `publish`, `search`, `recall`.
- CLI + seed/bench (на основе `benchmarks/searchserver`).

**v1:**
- каналы stable/testing/unstable как first-class; point-releases; `diff_releases`.
- `context(role)`; масштабирование read-репликами + балансировщик.
- аутентификация read/write; права на `publish`.

**v2:**
- интеграции с трекерами (GitHub Issues/Linear) как источник задач и приёмник
  результатов (шина, не начальник).
- политики хранения/архивации релизов; time-travel-запросы.
- наблюдаемость, дашборд публикаций.

## 13. Риски и ограничения

- R-1. **Сползание в PM-инструмент.** Митигируется §4 (не-цели) и релизной моделью:
  дистрибутив раздаёт правду, а не управляет проектом.
- R-2. **Стоимость эмбеддингов при ингесте** (BYO) — вне сервера, но влияет на время
  публикации; батчить на write-стороне.
- R-3. **Размер истории релизов** — политика хранения K версий / content-addressing.
- R-4. **Лаг публикации** для сценариев, требующих мгновенной свежести — покрывается
  каналом `testing`/`unstable`.

## 14. Метрики успеха

- Воспроизводимость: 100% идентичных результатов при повторных запросах к одному
  релизу.
- Грундинг: доля задач, где агент, пиновавшийся на релиз, дал корректный по текущей
  правде результат (в т.ч. на нишевых API — эталонный кейс PureBasic).
- Производительность: целевой rps/узел и латентность под нагрузкой (см. NFR-1).
- Принятие: число команд/проектов, использующих сервер как единую точку правды.
